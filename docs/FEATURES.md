# Running Art — 功能與開發歷程

> 用跑步軌跡畫圖：在地圖上手繪圖案，程式把它對齊到真實道路，
> 產生一條可以實際去跑的路線，並輸出 GPX。

**線上版**：https://justin121710.github.io/running-art/
**GitHub**：https://github.com/justin121710/running-art

---

## 一、目前的兩種形態

| | 桌面版 | 瀏覽器版 |
|---|---|---|
| 進入點 | `GPS_ART_vFINAL.py`（Tkinter） | `deploy/index.html` |
| 演算法 | `gpsart/` | **同一份** `gpsart/`，用 Pyodide 在瀏覽器執行 |
| 地圖來源 | osmnx 下載 | 瀏覽器 `fetch` 直接打 Overpass |
| 需要伺服器 | 否（本機執行） | **否**（純靜態網頁，免費空間即可） |
| 狀態 | 功能完整，仍是主力 | 核心流程已可用 |

兩者共用同一套演算法，這是刻意的設計：瀏覽器版最終要取代桌面版，
但演算法（那些花很多輪才調好的路徑邏輯）不重寫。

---

## 二、核心流程

```
框選範圍 → 下載路網 → 手繪圖案 → 設定起訖點 → 生成路線 → 匯出 GPX
```

1. **框選下載**：在地圖上拉一個方框，即時估算面積與下載時間
2. **前處理**：剪除死路 → 角度簡化 → 只保留最大連通塊 → 計算道路成本
3. **對齊道路**：把手繪線切段，每段在「走廊」內找最合適的道路
4. **路徑清理**：去尖刺、去迴圈、截彎取直、剪除折返、修補斷點
5. **輸出**：GPX（可匯入 Strava／Garmin）、PNG（桌面版）

## 三、路線品質：解決過的問題

這些是實際跑出來發現、逐一修掉的。**每一項都有對應的參數可調**（見 `gpsart/config.py`）。

### 3.1 切西瓜（路徑直線穿過建築物）

**成因**：後製步驟（去迴圈、剪枝、截彎取直）會刪掉中間節點，
使前後兩個節點之間根本沒有道路相連，畫出來就是一條直線硬連過去。

**解法**：`bridge_node_path_gaps()` — 找出「前後兩點沒有道路相連」的跳點，
沿真實道路重新接回來。只插入節點、不刪除，所以不會影響覆蓋率。

> 實測：997 公尺的直線切西瓜 → 消除，改成沿道路繞行（最長直線降到 137m）

另一個相關修正：繪圖時改用 `road_polyline_from_nodes()`，
它會展開道路的實際彎曲幾何，而不是把兩個節點用直線相連。

### 3.2 無意義折返（原地來回）

**成因**：既有的清理只抓「完全相同的節點」或「三點小尖刺」，
抓不到「往前走 100 公尺再原路折返回來」這種較長的來回。

**解法**：`remove_backtracks()` — 若實際走了 ≥80m 卻回到出發點 30m 內，判定為折返並剪掉。

**關鍵防呆**：畫圓也會「回到出發點」，天真的做法會把整個圓刪光。
所以用「細長度」（圍出的面積 ÷ 同周長圓的面積）分辨：
原路折返 ≈ 0（剪掉），繞一圈 ≈ 1.0（保留）。

> 實測：折返 2883m → 1342m；圓形保留 12/12 扇區，半徑誤差反而變小

### 3.3 過馬路繞遠路

**成因（兩個疊加）**：
1. 行人穿越道在圖資裡也是 `footway`，被人行道的懲罰倍率一起罰到
2. 走過的路被設成 `travel_weight = 1e9`（等於封死），路徑繞回附近時無路可走

**解法**：
- 下載時多抓 `footway` 子標籤，才分得出「人行道」與「行人穿越道」；穿越道不罰
- 「已走過」改成有限倍率 `REUSE_PENALTY_FACTOR = 4.0`

> 實測（畫圓，圓周 2827m）：6507m → 4606m（短 29%），且更貼合圓形

### 3.4 平行道路橫跳

**成因**：步行路網每條街兩側各有一條人行道（實測占全部邊的 44%），
加上每 40 公尺重新找一次最近節點，手抖幾公尺就會在兩側之間翻面。

**解法**：
- `smooth_drawn_stroke()` 先輕度平滑手繪線，消除手抖
- 提高人行道成本 `FOOTWAY_PENALTY = 2.5`，偏好車道中心線（一條街只剩一條線）

> 實測：橫跳 51 → 39 次。另外發現「取樣間距」是更有效的槓桿
> （40m→60m 可再降到 33 次且不犧牲貼合度），但預設仍保留 40m

### 3.5 閉合圖形被整圈抄掉 ⚠️ 最嚴重

**成因**：`advanced_shortcut_optimizer` 用「每節點間隔約 5 公尺」估算路徑長度。
在**閉合圖形**上，50 個節點之後的點在空間上就在起點旁邊，
演算法誤以為找到超划算的捷徑，**把整個圓抄掉**。

> 實測：58 個節點的圓被縮成 **1 個節點**

**解法**：單次捷徑最多只能跳過整條路徑的 1/4（`span_cap`）。
截彎取直是要修掉局部繞路，不該把使用者畫的形狀本身抹掉。

**這個修正一度沒有測試守護**（2026-07-23 補上）。原本的 `advanced_shortcut` 指標
餵的是兩點間的最短路徑，本來就沒有彎可截，輸入輸出都是 54 ——
`span_cap` 改壞了也看不出來。新增的 `ring_shortcut` 才測得到，
而且**半徑要選對**：崩塌只在「路徑長度和 `lookahead` 相當」時發生。

> 實測（fixture 路網，`lookahead=50`，與 `solve_route` 內部一致）：
> 220m 的圓接成 50 節點 → **無 `span_cap` 被抄成 1 個，有 `span_cap` 完整保留 50**。
> 380m 的圓是 110 節點，兩版都輸出 85 —— 太長，`span_cap` 咬不到，測不出差別。

### 3.6 起終點會跑去接筆畫的頭尾

**成因**：瀏覽器版接引擎時，漏掉了桌面版原本就有的兩個前處理。

**解法**：生成前先做
- `stitch_nearby_strokes()` — 斷開／交錯但很近的筆畫視為同一筆
- `reorder_closed_strokes()` — 閉合圖形旋轉起點，從離使用者起點最近處開始

兩者的門檻都改成**依整體圖形大小等比例**（整體對角線的 4%，夾在 15～120m），
小圖不會被亂縫、大圖也縫得起來。

> 實測：起點到路線起點 397m → **0m**；縫合門檻 30/100/200m 對 20/60/150m 的缺口行為完全正確

### 3.7 刪掉節點後路線會「原封不動長回來」

演算法再怎麼調，總有幾段是使用者自己看了不喜歡的，所以兩個版本都提供手動刪節點。

**成因**：刪掉節點後要沿道路把缺口接回來，但最短路徑當然會再次穿過剛剛刪掉的那個點，
看起來就像沒刪成功。

**解法**：被刪的節點進入 **banned 名單**，重接時用排除這些節點的子圖
（`bridge_node_path_gaps(Gw.subgraph(...))`），才會真的繞過去。
另外兩道防呆：這一段剩不到 2 個點就整批跳過；繞不過去而留下斷線就**整批自動復原**
並告知使用者，不會留下一條切過建築物的直線。

瀏覽器版的實作在 `engine.js` 的 Python 膠水裡（`_edit_delete` / `_edit_undo`），
不在 `gpsart/` —— 它用的是既有的 `bridge_node_path_gaps()`，沒有動演算法核心。

> 實測（40m 網格合成路網，方形路線 1443m / 37 節點）：
> 刪掉一邊中段 3 個節點 → 1523m（+80m，正好是繞過一個街廓的代價）、
> 被刪節點確認沒有回到路線中、相鄰點最大跳躍維持 40m（無切西瓜）、復原精準還原。
> 刪掉整條邊 8 個節點 → 整段平移到隔壁街廓，長度不變（曼哈頓網格上等長路徑，符合預期）。

### 3.8 方向箭頭沒有貼著路線

路線預覽的方向箭頭原本用兩個寫錯的地方疊在一起，看起來歪歪的甚至指反：

1. **拿經緯度差算角度** —— Mercator 投影橫向會被 `1/cos(緯度)` 拉伸，
   在台灣緯度大約差 10%，角度就歪掉。要先 `map.project()` 成平面像素再算。
2. **`90 - 角度` 符號錯** —— `atan2(dy,dx)` 算出來已經是螢幕角度（y 軸向下、順時針為正），
   而箭頭字元 ➤ 預設就指向右，直接丟給 CSS `rotate()` 即可；
   寫成 `90-角度` 會得到沿正確方向的**鏡像**。

> 實測（合成網格的方形路線，13 個箭頭）：修正後與螢幕實際方向誤差 **0 度**，
> 舊寫法最大誤差 **180 度**（完全反向）。

## 四、介面

### 桌面版
Tkinter。功能完整但介面傳統，將被瀏覽器版取代。
含「路徑微調手術台」（`RouteEditor`）：可框選節點刪除，刪除後會自動繞過該節點沿道路重接。

### 瀏覽器版（深色、手機優先）

- **全螢幕地圖 + 底部工具列**：拇指可及，地圖畫面最大化
- **輕量引導列**：依狀態自動推進（不用按「下一步」），新手有指引、老手不被擋
- **模式切換**：只有「移動」模式能拖曳地圖，畫圖時鎖住，不會誤觸
- **已下載範圍**：青色虛線框 + 框外變暗，並限制只能在框內畫圖／設起訖點
- **結果面板**：距離、與圖形吻合度、預估時間
- **可調配速**：−／+ 調整（3:00～15:00），時間即時更新並記住設定
- **路線預覽拉桿**：拖曳或播放，顯示行進位置與方向箭頭
- **微調路徑**：與桌面版同一套邏輯（見 3.7）
- **生成後鎖定**：生成後不能再誤動圖案／起訖點，要按「修改圖案」（保留紅線續改）
  或「重新繪製」（清空重來）才解鎖
- **反轉方向**：整條路線首尾對調，起訖點跟著換
- **匯出圖片**：底圖＋路線的 PNG，路線置中、資訊欄在下（比例尺＋距離／配速／時間）。
  - 三種社群比例：**1:1**（正方形）、**4:5**（直式貼文）、**9:16**（限時動態）
  - 圖層可勾選：路線／方向箭頭／手繪紅線／起訖點
  - 方向箭頭用亮黃色、沿整條路線（含連接虛線）標示前進方向
  - 底圖比 App 內再提亮一些，道路看得清楚
